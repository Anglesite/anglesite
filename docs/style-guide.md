# Style Guide

Design principles and coding standards for HTML, CSS, and TypeScript generated by Anglesite. This guide codifies existing patterns so that new pages and components stay consistent with the rest of the site. For design token values, color palettes, and layout patterns, see `template/docs/design-system.md`.

## Design principles

1. **Reuse before you create.** When a visual pattern already exists — cards, grids, form fields, buttons — use the existing classes from `global.css` or extract a shared component. Never duplicate markup and styles across pages.
2. **Design tokens over magic numbers.** Use CSS custom properties (`--space-md`, `--color-primary`, etc.) for all colors, spacing, font sizes, radii, and shadows. Never hardcode values.
3. **Semantic HTML first.** Choose the right element (`<section>`, `<nav>`, `<article>`, `<time>`) before reaching for classes. Use ARIA only when native semantics are insufficient.
4. **Zero client-side JS by default.** Standard pages ship no JavaScript. Add it only when progressive enhancement requires it (search, filters, embeds), and always provide a `<noscript>` fallback.
5. **Mobile-first, minimal breakpoints.** The `body { max-width: 48rem }` constraint handles most layout. Add breakpoints only when content genuinely needs a different arrangement at wider viewports.

## HTML

### Document Structure

Every page rendered with `BaseLayout` follows this landmark pattern:

```html
<html lang="en">
  <body>
    <a href="#main" class="skip-link">Skip to content</a>
    <header class="h-card" data-pagefind-ignore>...</header>
    <main id="main" data-pagefind-body>...</main>
    <footer data-pagefind-ignore>...</footer>
  </body>
</html>
```

- The skip link is always the first focusable element.
- `id="main"` on `<main>` targets the skip link.
- `data-pagefind-body` / `data-pagefind-ignore` control search indexing.

### Heading Hierarchy

- One `<h1>` per page, matching the page title.
- Section headings use `<h2>`, sub-section headings use `<h3>`. Never skip levels.
- Wire headings to their parent sections: `<section aria-labelledby="heading-{id}">`.

### Accessibility Minimums

| Pattern | Requirement |
|---|---|
| Images | `alt` text describing the image. Decorative images: `alt="" aria-hidden="true"`. |
| Links / buttons | Visible label or `aria-label`. |
| Focus | `a:focus-visible` outline via `global.css`. Never remove focus indicators. |
| Forms | Every `<input>` paired with `<label for="...">`. Use `autocomplete` on personal data fields. |
| Touch targets | Minimum 44×44px on mobile. Add padding if the element itself is smaller. |
| Time | `<time datetime="ISO-8601">human text</time>`. |
| Color contrast | 4.5:1 for body text, 3:1 for large text (WCAG AA). |
| Color independence | Never convey information through color alone — pair with text, icons, or patterns. |
| Motion | Respect `prefers-reduced-motion: reduce`. |
| Lazy loading | Images below the fold: `loading="lazy"` with explicit `width` and `height`. |
| ARIA | Use native HTML semantics first. Add `aria-label`, `aria-expanded`, `aria-selected` only when native semantics don't communicate the state. |

### IndieWeb Microformats

The site uses microformats2 markup. Maintain these when editing:

| Class | Element | Purpose |
|---|---|---|
| `.h-card` | `<header>` | Site/author identity |
| `.p-name` | heading or `<span>` | Name within h-card or h-entry |
| `.u-url` | `<a>` | Canonical URL |
| `.h-entry` | `<article>` | Blog post container |
| `.dt-published` | `<time datetime="">` | Publication date |
| `.p-summary` | `<p>` | Post excerpt |
| `.e-content` | `<div>` | Post body |
| `.p-category` | `<li>` or `<span>` | Tags/categories |
| `.u-syndication` | `<a rel="syndication">` | Cross-posted URLs |

### Forms

```html
<form method="POST" action={url}>
  <div class="form-field">
    <label for="name">Name</label>
    <input type="text" id="name" name="name" required autocomplete="name" />
  </div>
  <button type="submit">Submit</button>
</form>
```

- Always pair `for`/`id` on label and input.
- Always set `autocomplete` on personal data fields.
- Always use `required` for mandatory fields (HTML5 validation).
- Always use explicit `type="submit"` on buttons.

### JSON-LD Structured Data

Build JSON-LD as a typed object in the frontmatter, pass it to the layout via the `jsonLd` prop, and render it in `<head>`:

```html
<script is:inline type="application/ld+json" set:html={JSON.stringify(jsonLd)} />
```

Never inline the JSON directly in the template.

## CSS

### Architecture

Vanilla CSS with custom properties (see [ADR-0004](decisions/0004-vanilla-css-custom-properties.md)). No Tailwind, Sass, or PostCSS.

Four CSS files exist:

| File | Scope | Imported by |
|---|---|---|
| `global.css` | Sitewide tokens, element styles, reusable classes | `BaseLayout` (`is:global`) |
| `menu.css` | Menu page styles (`menu-*` prefix) | Menu pages (`@import`) |
| `kiosk.css` | Kiosk/mobile menu styles (`kiosk-*` prefix) | `KioskLayout` (`@import`) |
| `immersive.css` | Full-viewport canvas styles (`immersive-*` prefix) | `ImmersiveLayout` (`@import`) |

### Linting

The template includes [Stylelint](https://stylelint.io/) with [`stylelint-declaration-strict-value`](https://github.com/AndyOGo/stylelint-declaration-strict-value) to enforce design token usage. Run it with:

```sh
npm run lint:css
```

The linter flags raw values for `color`, `background`, `font-size`, `border-radius`, and `box-shadow` that should use `var(--token)` instead. Allowed raw values: `inherit`, `none`, `transparent`, `currentColor`, `auto`, `0`, `#fff`, `#000`, `white`, `black`, `50%`.

To disable the rule for a block with a documented reason (e.g., print styles, semantic color pairs):

```css
/* stylelint-disable scale-unlimited/declaration-strict-value -- reason */
.example { color: #custom; }
/* stylelint-enable scale-unlimited/declaration-strict-value */
```

### When to Use Global vs. Scoped Styles

- **Global CSS** (`global.css`): element-level resets, design tokens, and reusable component classes (`.post-list`, `.form-field`, `.testimonial-card`, `.buy-button`).
- **Feature CSS files** (`menu.css`, `kiosk.css`, `immersive.css`): styles specific to a feature that has its own layout or namespace.
- **Scoped `<style>` blocks**: page-specific styles that are not reusable (e.g., theming a third-party widget). Prefer extracting to a CSS file if the styles grow beyond ~20 rules.

**Default to global.css** for anything used on more than one page. Astro's scoped styles add specificity and can make overrides harder. Use them sparingly.

### Design Tokens

All tokens live in `global.css` under `:root`. The design interview generates `src/design/tokens.css` which overrides the defaults. Use them everywhere:

```css
/* Colors */
--color-primary   --color-accent   --color-bg
--color-text      --color-muted    --color-surface   --color-border

/* Typography */
--font-heading    --font-body
--font-size-sm    --font-size-base  --font-size-lg
--font-size-xl    --font-size-2xl   --font-size-3xl   --font-size-4xl

/* Spacing */
--space-xs  --space-sm  --space-md  --space-lg  --space-xl

/* Layout */
--max-width

/* Radii */
--radius-sm  --radius-md  --radius-lg

/* Shadows */
--shadow-sm  --shadow-md
```

If you need a value that doesn't exist as a token, derive it from existing tokens using `calc()`:

```css
/* Good: derived from tokens */
padding: calc(var(--space-sm) + var(--space-xs));

/* Bad: magic number */
padding: 0.75rem;
```

To add a new token, add it to `global.css` under `:root` and document it here. Feature-specific tokens use a namespace prefix (`--menu-*`, `--kiosk-*`).

### Class Naming

Three tiers, chosen by scope:

| Scope | Convention | Examples |
|---|---|---|
| Global / reusable | Flat descriptive name | `post-list`, `form-field`, `buy-button`, `skip-link` |
| Feature-namespaced | `feature-element` | `menu-nav`, `menu-section`, `menu-item` |
| Complex component (BEM) | `block__element--modifier` | `kiosk-topbar__logo`, `kiosk-tabs__tab`, `immersive-overlay--visible` |

- **State classes** applied by JavaScript use the `is-` prefix: `is-active`, `is-visible`.
- **Data-attribute styling** is preferred for variant states: `[data-diet="vegetarian"]`, `[aria-selected="true"]`.
- Microformat classes (`.h-card`, `.h-entry`, `.p-name`, `.u-url`) follow IndieWeb conventions — don't rename them.

### Breakpoints

Only two breakpoints are in use. Add new ones only when the content demands it:

| Breakpoint | Where used | Purpose |
|---|---|---|
| `min-width: 640px` | `kiosk.css` | Slightly larger text on wider mobile |
| `min-width: 48rem` | `menu.css` | Two-column grid for menu items |

All breakpoints are `min-width` (mobile-first). The main site layout needs no breakpoints because `body { max-width: 48rem }` constrains the content width. Use `rem` for breakpoint values, not `px`. Grids should use `auto-fill` with `minmax()` for fluid column counts.

### Required Media Queries

Every CSS file that includes animations or transitions must include:

```css
@media (prefers-reduced-motion: reduce) {
  /* disable or reduce animations */
}
```

Include `@media print` rules when the page has content users might print (menus, contact info, location):

- Hide navigation and decorative elements
- Show link URLs inline: `a[href]::after { content: " (" attr(href) ")"; }`
- Remove backgrounds and shadows to save ink

### Theme Variants

Dark mode and kiosk mode use `[data-theme]` attribute selectors on `:root`:

```css
:root[data-theme="dark"] {
  --color-bg: #0a0a0a;
  --color-text: #e5e5e5;
}
```

Never use a separate stylesheet for themes — override tokens on the same custom properties so all components adapt automatically.

### Button Styles

All interactive buttons share this pattern:

```css
background: var(--color-primary);
color: #fff;
font-weight: 600;
border: none;
border-radius: var(--radius-md);
cursor: pointer;
```

Hover: `opacity: 0.9`. Focus: `outline: 2px solid var(--color-primary); outline-offset: 2px`.

## TypeScript

### Strict Mode

All TypeScript extends `astro/tsconfigs/strict`. This means:

- `strict: true` (includes `noImplicitAny`, `strictNullChecks`, etc.)
- No `any` types without explicit justification
- All function parameters and return types are inferrable or annotated

### Astro Component Frontmatter

Every component declares a local `Props` interface:

```typescript
interface Props {
  title: string;
  description: string;
  image?: string;
  href?: string;
}
const { title, description, image, href } = Astro.props;
```

- Use `interface Props` (not `type Props`) — Astro convention.
- Declare `Props` locally in the file, not imported from elsewhere.
- Required props have no default. Optional props use `?` with destructuring defaults: `const { label = "Buy" } = Astro.props`.
- Use union types for constrained values: `style: 'inline' | 'floating' | 'button'`.
- Destructure `Astro.props` in the frontmatter, not in the template.

### Script Files (`scripts/*.ts`)

1. Start with a JSDoc module comment explaining the file's purpose.
2. Use named exports only — no default exports.
3. Prefer pure functions where possible.
4. Export interfaces for data shapes, types for unions.
5. Add JSDoc (`/** ... */`) before every exported function and interface.

### Content Collections

Content lives in `src/content/` with schemas defined in `src/content.config.ts` using Zod:

```typescript
const posts = defineCollection({
  type: "content",
  schema: z.object({
    title: z.string(),
    description: z.string(),
    publishDate: z.string().transform((str) => new Date(str)),
    tags: z.array(z.string()).default([]),
    draft: z.boolean().default(false),
  }),
});
```

- Every content collection has a Zod schema — no unvalidated frontmatter.
- Keystatic schema (`keystatic.config.ts`) mirrors the Zod schema so the CMS editor stays in sync.
- Filter drafts at query time: `getCollection("posts", ({ data }) => !data.draft)`.
- Prefer content collections over loose data files in `src/data/` — collections get type-safe querying and Hot Module Replacement.

### Config Reading

Pages that read `.site-config` use this pattern:

```typescript
import { readFileSync, existsSync } from "node:fs";
import { resolve } from "node:path";

const configPath = resolve(process.cwd(), ".site-config");
const config = existsSync(configPath) ? readFileSync(configPath, "utf-8") : "";
const value = config.match(/^KEY=(.+)$/m)?.[1]?.trim() ?? "fallback";
```

### Data Files

Place shared data in `src/data/*.ts`. Export typed arrays or objects:

```typescript
export interface WebsiteType {
  slug: string;
  title: string;
  description: string;
}

export const websiteTypes: WebsiteType[] = [
  { slug: "restaurant", title: "Restaurant", description: "..." },
];
```

### API Routes

API routes (`.ts` files in `src/pages/`) follow Astro conventions:

```typescript
export async function GET(context: APIContext) {
  // ...
}

export async function getStaticPaths() {
  // ...
}
```

- Export named HTTP method handlers (`GET`, `POST`), not default exports.
- Use `APIContext` type from Astro for the parameter.
- Static routes use `getStaticPaths()` for dynamic segments.

## Components

### When to Create a Shared Component

Extract a component when:

- The same visual pattern appears on **two or more pages** (e.g., card grids, buttons, badges).
- A block of markup has **its own props** (e.g., a booking widget that takes a provider and username).
- The markup exceeds ~20 lines and has internal structure worth naming.

Do not extract when:

- The markup is only used once and is simple enough to inline.
- The "component" would just be a thin wrapper with no props or logic.

Place reusable components in `src/components/`. Name them in PascalCase matching their purpose: `TileCard.astro`, `BuyButton.astro`, `BookingWidget.astro`.

### Component File Structure

```text
---
// Imports
// Props interface
// Props destructuring
// Logic
---
<!-- Template HTML -->

<style>
  /* Scoped styles (only if needed) */
</style>

<script>
  /* Client JS (only if needed) */
</script>
```

### Layout Selection

| Layout | When to use |
|---|---|
| `BaseLayout` | All standard content pages |
| `ImmersiveLayout` | Full-viewport canvas, art, or interactive experiments |
| `KioskLayout` | Mobile-first kiosk displays (QR/NFC menus) |

## Markdown

### Linting

The template includes [markdownlint-cli2](https://github.com/DavidAnson/markdownlint-cli2) for documentation quality. Run it with:

```sh
npm run lint:md
```

### Rules

The config (`.markdownlint.jsonc`) disables rules that conflict with project conventions:

| Disabled rule | Why |
|---|---|
| MD013 (line-length) | Docs use long lines for readability in editors |
| MD024 (no-duplicate-heading) | Same heading text is valid in different sections |
| MD036 (no-emphasis-as-heading) | Bold text as sub-heading is intentional in some docs |
| MD060 (table-column-style) | Compact table pipes are project convention |

All other rules are enforced. Key ones:

- **MD032** — Blank lines around lists (prevents rendering bugs)
- **MD022** — Blank lines around headings
- **MD031** — Blank lines around fenced code blocks
- **MD040** — Fenced code blocks must specify a language (`text` for plain output)
- **MD034** — No bare URLs (use `<url>` or `[text](url)`)

## Anti-Patterns

These patterns have caused problems before. Avoid them:

| Anti-pattern | Do this instead |
|---|---|
| Duplicating card/grid CSS across pages | Reuse classes from `global.css` or extract a shared component |
| Hardcoding colors, spacing, or font sizes | Use design tokens (`--color-*`, `--space-*`, `--font-size-*`) |
| Adding `<div>` soup for layout | Use semantic elements (`<section>`, `<nav>`, `<article>`) |
| Desktop-first media queries (`max-width`) | Mobile-first (`min-width`) |
| Adding client-side JS for styling | Use CSS (`:hover`, `:focus-visible`, `[aria-selected]`) |
| Importing a CSS framework | Vanilla CSS with custom properties (see ADR-0004) |
| Platform-specific commands (`sips`, `open`, `sed -i ""`) | Cross-platform alternatives via `scripts/platform.ts` |
