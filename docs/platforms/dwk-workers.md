# `@dwk/*` workers — self-owned IndieWeb endpoints

The [`@dwk/*`](https://github.com/davidwkeith/workers) packages implement open web standards as composable, `fetch`-compatible Cloudflare Worker handlers that deploy to the **owner's own** Cloudflare account. They are the self-owned alternative to third-party IndieWeb hosts (webmention.io, indieauth.com): identity, tokens, and data stay on infrastructure the owner controls. Anglesite is the project's first consumer.

Reference for the webmaster agent — read when an owner wants the **active** IndieWeb (live endpoints), not just the passive microformats/`rel="me"`/RSS that already ship. Not user-facing documentation. The decision is recorded in [ADR-0020](../decisions/0020-active-indieweb.md); the full design is in `docs/superpowers/specs/2026-06-09-active-indieweb-design.md`.

## Status

> **Unpublished.** Every `@dwk/*` package is at `0.0.0` and not on npm yet. `/anglesite:indieweb` is written publish-ready (it installs from npm) and stops gracefully — "this feature isn't available yet" — until the packages ship. Do not attempt git installs or forks.

## When to recommend it

- The owner wants to **sign in to IndieWeb services with their domain** (IndieAuth), **receive cross-site mentions/replies** (Webmention), or **publish from a Micropub client** (e.g. a phone app) — and wants to own the endpoints rather than delegate them
- The owner already controls their domain and Cloudflare account (the Anglesite default)
- The audience or the owner is IndieWeb-aware (developer blogs, IndieWeb, open-source projects)
- The owner explicitly wants to avoid third-party identity/mention hosts

## When **not** to recommend it

- The site is still on a `*.workers.dev` address — identity must be HTTPS-rooted at a **custom primary domain**; the skill refuses otherwise
- The owner is a typical non-technical SMB with no IndieWeb need — the passive IndieWeb (already shipped) is enough; don't add live endpoints they won't use
- The owner wants comments specifically — point them at `/anglesite:giscus` instead
- Don't offer this during `/anglesite:start`; it's an opt-in enhancement for sites that have been running a while

## The MVP cohort

| Package | Endpoint | What it does |
|---|---|---|
| `@dwk/indieauth` | `/auth`, `/auth/token`, metadata | Authorization + token + metadata endpoints; PKCE; `rel=me` profile verification; issues DPoP-bound, scoped access tokens |
| `@dwk/webmention` | `/webmention` | Receiver (sync-validates target, async-verifies the source link via a Queue, stores to a D1 inbox) + sender (notifies targets on publish) |
| `@dwk/micropub` | `/micropub`, `/media` | Create/update/delete posts; JSON + form-encoded; R2-backed media endpoint; `q=config`/`q=source`; consumes IndieAuth tokens |

Later cohorts (separate skills/specs, not in the MVP): Microsub, WebSub, WebFinger, host-meta, `did:web`/VC, Solid Pod, WebAuthn.

## How it works

1. Each package exports a factory (`createIndieAuth(config)`, etc.) returning a handler mountable under a path prefix.
2. `/anglesite:indieweb` composes the chosen handlers into the site's existing `site-entry.js` Worker, ahead of the static `ASSETS` fallthrough — the same per-binding gating pattern that membership, A/B, and consent already use. One Worker, one domain, one deploy.
3. Identity is rooted at the primary domain: the same domain serves the page (`me=https://example.com`), the discovery `<link>`s, and the endpoints.

## Bindings each package needs

The composed Worker's `Env` is the union of the mounted packages' fragments; a missing binding fails loudly at startup (no silent degradation).

| Binding | Type | Package | Purpose |
|---|---|---|---|
| `AUTH_DB` | D1 | indieauth | Authorization codes + issued tokens (strongly-consistent; never KV) |
| `WEBAUTHN` | Durable Object | webauthn | Passkey challenge state + credential records (owner authentication) |
| `OWNER_AUTH_DB` | D1 | (anglesite) | Single-use backup-code hashes (owner recovery) |
| `INDIEAUTH_SESSION_KEY` | secret | (anglesite) | Signs the owner-session cookie after a passkey/backup-code sign-in |
| `INDIEWEB_REG_TOKEN` | secret | (anglesite) | One-time gate for the first passkey enrolment |
| `MICROPUB_DB` | D1 | micropub | Published post records (mf2 source) + DPoP `jti` replay store |
| `WEBMENTION_INBOX` | D1 | webmention | Verified-mention inbox (stores `{ source, target, verified_at }` only — no author/content) |
| `MEDIA` | R2 | micropub | Media-endpoint blob storage |
| `WEBMENTION_QUEUE` | Queue | webmention | Async source-link verification |
| `SITE_URL` | var | webmention | The site's own origin — `@dwk/webmention`'s `baseUrl` at construction (the queue consumer runs off-request) |
| `INDIEAUTH_SIGNING_KEY` | secret | indieauth | Token signing material (generated by the skill) |
| `GITHUB_TOKEN` | secret | bridge | Fine-grained PAT, `contents:write` — for the Micropub `.mdoc` commits |

Provision D1/R2 with the Cloudflare MCP tools (`mcp__cloudflare__d1_database_create`, `mcp__cloudflare__r2_bucket_create`) so the resource id can be written straight into `wrangler.jsonc`; Queues use `npx wrangler queues create`; secrets use `npx wrangler secret put`.

## Micropub → static content (the publish loop)

`@dwk/micropub` stores posts in D1 — its storage is **not** pluggable (only the post-URL policy is). To make a Micropub post a first-class, owner-editable Keystatic page, Anglesite bridges D1 to git:

1. `POST /micropub` → the handler writes the post to `MICROPUB_DB` and returns `201 Location: /notes/<slug>`.
2. A worker bridge (queue/cron) materializes the new record into `src/content/notes/<slug>.mdoc` and commits it via the GitHub API using `GITHUB_TOKEN`.
3. The commit triggers the **GitHub Actions deploy workflow** (`template/.github/workflows/deploy.yml`), which rebuilds and `wrangler deploy`s the static site.
4. `/notes/<slug>` is live after the rebuild (~1–2 min). `q=source`/`update`/`delete` are instant against D1; the rendered page is the eventually-consistent public copy.

The deploy workflow is **self-gating** — it ships in the template but no-ops until `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` are set as repo secrets (the skill sets them; or `gh secret set` by hand). Sites that deploy locally with `/anglesite:deploy` are unaffected.

## DPoP everywhere

`@dwk/micropub` requires a DPoP proof on **every** authenticated request and binds the token's subject to the configured `me`. This is stricter than the OAuth/Micropub baseline and is deliberate (a stolen bearer token alone is never sufficient). The practical consequence: **bearer-only Micropub clients — including micropub.rocks' default flow — cannot authenticate.** Only DPoP-capable clients work. Tell the owner which clients are supported rather than letting them discover this as a failure.

## Cost

Free within Cloudflare's free tier for a small site (Workers, D1, R2, Queues all have generous free allowances). No third-party service fees.

## Privacy & security

- Identity, tokens, and all mention/post data live only on the owner's Cloudflare account — nothing leaves it
- `INDIEAUTH_SIGNING_KEY` and `GITHUB_TOKEN` are secret bindings (wrangler/GitHub secrets), never in source; `/anglesite:deploy`'s token scan confirms this
- `/auth`, `/micropub`, `/media`, `/webmention` are intentional public endpoints — the deploy scans are configured to allow them
- The `GITHUB_TOKEN` is least-privilege: fine-grained, single-repo, `contents:write` only
- Privacy policy should mention that the site receives and stores Webmentions (data from other sites and their authors)

## Relationship to the passive IndieWeb

This does **not** replace the microformats / `rel="me"` / RSS / POSSE that every Anglesite site already ships (ADR-0006). It adds the live endpoints on top. See `template/docs/indieweb.md` for the full picture.
